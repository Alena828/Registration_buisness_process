# Use_cases_diagram

Данный раздел содержит Use-cases диаграммы для процеесса регистрации собаки в Реестре с участием следующих акторов: Заявитель, Консультант, Сотрудник реестра, Администратор вет.клиники и Ветеринар. Каждый уровень диаграммы представлен отдельной диаграммой.


## UC-1.png: Зарегистрировать собаку в Реестре

**Акторы**: Заявитель, Консультант, Администратор ветклиники, Ветеринар, Сотрудник реестра  
**Триггер**: Поступление заявления от владельца собаки на регистрацию в реестре.

### Основной сценарий

1. Заявитель обращается за консультацией по процедуре регистрации.
2. Консультант информирует о порядке регистрации, необходимых документах и предоставляет список ветклиник.
3. Заявитель записывается в ветклинику через консультанта.
4. Администратор ветклиники фиксирует запись и согласовывает время.
5. Заявитель посещает ветклинику, предоставляет документы.
6. Ветеринар и сотрудник реестра проводят регистрацию:
    - Проверка заявления, документов и данных.
    - Заключение договора.
    - Оформление ветпаспорта.
    - Внесение данных в ветпаспорт.
    - Внесение данных в реестр.
7. Заявитель получает подтверждение регистрации.

### Альтернативный сценарий  - Изменение данных

1. Заявителю требуется изменить ранее внесённые данные в реестре.
2. Сотрудник реестра предоставляет анкету на изменение данных.
3. Заявитель заполняет анкету и предоставляет подтверждающие документы.
4. Сотрудник реестра вносит изменения и выдаёт справку об изменении.

## UC-2.png: Изменение данных в реестре 

**Акторы**: Сотрудник реестра, Заявитель  
**Триггер**: Необходимость корректировки ранее внесённых данных о собаке или владельце.

### Основной сценарий

1. Заявитель заполняет регистрационную анкету на изменение данных.
2. Заявитель предоставляет документы, подтверждающие необходимость изменений.
3. Сотрудник реестра проверяет корректность запроса.
4. Сотрудник реестра вносит изменения в Реестр.
5. Сотрудник реестра выдаёт справку об изменении в Реестре.


### Альтернативный сценарий: 
1.4.1 - отказ в изменении данных из-за несогласия заполнить анекту или из-за некорректного заполнения. 

### Альтернативный сценарий: 
1.4.1 - отказ в изменении данных из-за отсутствия текущей регистрации собаки. 

### Альтернативный сценарий: 
1.4.2 - отказ в изменении данных из-за отсутсвия корректного документа, подтверждающего необходимость изменений.

## UC-3.png: Проконсультировать по процедуре регистрации

**Акторы**: Консультант, Заявитель, Сотрудник реестра
**Триггер**: Обращение заявителя за разъяснениями.

### Основной сценарий

1. Консультант информирует о порядке регистрации.
2. Консультант информирует о необходимых документах.
3. Консультант предоставляет список и контактную информацию ветклиник.
4. По запросу консультанта сотрудник реестар проверяет наличие текущей регистрации у собаки.
5. Консультант выдаёт памятку о порядке прохождения регистрации.


### Альтернативный сценарий

1. Консультант информирует о порядке регистрации.
2. Консультант информирует о необходимых документах.
3. Консультант предоставляет список и контактную информацию ветклиник.
4. По запросу консультанта сотрудник реестар проверяет наличие текущей регистрации у собаки.
5. У собаки уже есть текущая регистрация.
6. Заявитель перенаправляется нна процесс изменения данных регистрации.

## UC-3.png: Записаться в ветклинику через консультанта

**Акторы**: Заявитель, Консультант, Администратор  
**Триггер**: Необходимость выбора времени даты визита в ветклинику.

### Основной сценарий

1. Заявитель заполняет регистрационные данные через консультанта.
2. Консультант передаёт данные администратору.
3. Администратор согласовывает с заявителем время и дату приёма.
4. Администратор подтверждает запись.
5. Администратор вносит данные о приёме в систему.

### Альтернативный сценарий: 
Прекращение записи на 1.2.1.1 - пользователь отказался предоставлять данные

### Альтернативный сценарий: 
Прекращение записи на 1.2.1.3 - нет подходящего времени для записи

### Альтернативный сценарий:
На 1.2.1.4 не удалось подтвердить запись на прием - прделагать новое время записи.

## UC-5.png: Провести регистрацию (в составе UC-1)

**Акторы**: Ветеринар, Сотрудник реестра, Администратор ветклиники, Заявитель
**Триггер**: Успешная запись и прибытие заявителя в ветклинику.

### Основной сценарий

1. Сотрудник реестра и администратор проверяют заявление, документы и данные.
2. Администратор заключает договор с заявителем.
3. Ветеринар проводит вет.прием.
4. Ветеринар офрмляет ветпаспорт.
5. Администратор передает данные регистрации сотруднику реестра.
6. Сотрудник реестра вносит данные регистрации в реестр.

   
### Альтернативный сценарий:
1.3.1 - отказ в регистрации из-за существующей регистрации - направить на изменение данных регистрации

### Альтернативный сценарий: 
1.3.1 - отказ в регистрации из-за проблем с документами 

### Альтернативный сценарий: 
1.3.2 - отказ в записи из-за несоставления договора/ непрошедшей оплаты

### Альтернативный сценарий: 
1.3.6 - отказ в записи из-за неподходящего состояния здоровья собаки
